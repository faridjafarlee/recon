# Artifact templates

Copy-pasteable skeletons for the three output artifacts produced by the recon
system. Workers populate these; the orchestrator assembles and stores the
results under `.recon/`.

---

## ARCHITECTURE.md

```markdown
# Architecture map

> Generated by recon/mapper. Last run: <!-- ISO date -->

## <Area>

**Purpose:** What this area of the codebase is responsible for; its role in the
overall system.

**Entry points:** List the top-level files, NestJS module registrations,
GraphQL schema roots, or React router paths that are the public surface of
this area.

**Key files:**
- `path/to/file.ts` — one-line description of its role
- `path/to/other.ts` — one-line description

**Data flow:** Describe how a request or event enters this area, travels
through layers (controller → service → repository → database, or component →
hook → store), and exits.

**Dependencies:** External packages, other internal areas, or infrastructure
services (MongoDB, Redis, third-party APIs) this area relies on.

**Risk leads:** Observations that warrant deeper inspection by the hunter or
refactoring-auditor (not confirmed findings — those go in ISSUES.md).

---

<!-- repeat ## <Area> block for each area -->
```

## ISSUES.md

```markdown
# Issues

> Generated by recon/hunter. Last run: <!-- ISO date -->
> Format: see `${CLAUDE_PLUGIN_ROOT}/skills/recon/reference/conventions.md`.

## Findings

<!-- An area with no findings gets a "### <Area> — no findings" line here. -->

### <Title — short imperative phrase>

- **Category:** security | performance | reliability | tooling | test | refactoring
- **Severity:** critical | high | medium | low
- **Location:** `path/file:line` (+ related sites if any)
- **What it is:** One or two concrete sentences describing the issue.
- **Why it matters:** The real impact if left unaddressed.
- **Trigger / exploit:** The concrete code path or user action that hits it.
- **Proposed fix:** The minimal root-cause change required, described but not
  applied here.
- **Confidence:** high | medium | low — note what additional evidence would
  confirm or refute it.
- **Source:** <tag>
- **Status:** - [ ]

---

<!-- repeat ### block for each finding, ordered by severity descending -->

## Dismissed leads

- <Title> — <reason dismissed in one clause>

## Frontend findings (Playwright)
<!-- Browser-driven findings (Source: [frontend]); evidence under .recon/frontend/. Same entry format. -->

## Exploited vulnerabilities (active pentest)
<!-- Active-pentest results (Source: [pentest]); PoC + evidence under .recon/pentest/. Placed after Frontend findings. -->

### <Title — e.g. "BOLA: account A reads account B's orders (API1)">

- **Title:** short imperative phrase (name the OWASP API ID)
- **Category:** security
- **Severity:** critical | high | medium | low
- **Location:** `path/file:line` (the vulnerable handler)
- **What it is:** the vuln + its OWASP API ID.
- **Why it matters:** the proven impact.
- **Trigger / exploit:** the attack path that hits it (summarised; the full request is in PoC).
- **PoC:** the exact request that demonstrates it (+ redacted impact-proving response); evidence `.recon/pentest/<slug>.http` / `<slug>.log`.
- **Proposed fix:** the minimal root-cause change, described (not applied).
- **Confidence:** high (proven by PoC).
- **Source:** [pentest]
- **Status:** - [ ]

---

### Negative results
<!-- High-value hypotheses that were probed and held. -->
- <OWASP ID> — <probe> → <outcome> (defense confirmed)
```

```markdown
> Generated by recon/refactoring-auditor. Last run: <!-- ISO date -->
> Format: see `${CLAUDE_PLUGIN_ROOT}/skills/recon/reference/conventions.md`.
> Findings here are tagged refactoring or performance; security and reliability
> issues go in ISSUES.md.

## Findings

<!-- An area with no findings gets a "### <Area> — no findings" line here. -->

### <Title — short imperative phrase>

- **Category:** refactoring (OPTIMIZATIONS holds refactoring-auditor findings; the full Category enum lives in conventions.md)
- **Severity:** critical | high | medium | low
- **Location:** `path/file:line` (+ related sites if any)
- **What it is:** One or two concrete sentences describing the smell or
  inefficiency.
- **Why it matters:** Maintainability or performance impact.
- **Trigger / exploit:** The usage pattern or call path that hits the
  bottleneck or smell.
- **Proposed fix:** The specific refactoring technique (referencing
  `${CLAUDE_PLUGIN_ROOT}/skills/recon/knowledge/refactoring/refactoring-techniques.md`) or optimisation
  strategy recommended.
- **Confidence:** high | medium | low — note what would confirm it.
- **Source:** <tag>
- **Status:** - [ ]

---

<!-- repeat ### block for each finding, ordered by severity descending -->

## Dismissed leads

- <Title> — <reason dismissed in one clause>
```

## Fix dossier — `.recon/fixes/<finding-id>/fix.md`

```markdown
# Fix: <Finding title>

**Finding:** <id> — <category>/<severity> — `path/file:line` (see ISSUES.md)

## What's being fixed
<concrete description>

## Why it's needed
- Root cause: <…>
- Why it matters: <impact>
- Trigger / exploit: <…>

## Solution & why it's best
- Change: <what was changed>
- Alternatives considered: <option → why rejected> (one per line)
- Why this is minimal and root-cause: <…>

## Test
<reproduce-test path>  — or  `no automated test: <why>` + manual verification procedure

## Regression check
Full suite vs Phase-1 baseline: `<X/Y> → <X/Y>`

## Performance
- Target: <micro-bench | bench-cmd | suite | n/a>
- Before: <mean> ± <spread> (<perfRuns> runs)
- After:  <mean> ± <spread> (<perfRuns> runs)
- Change: <±%>  <regression flag if applicable>

## Verification
Verifier verdict: <APPROVE | APPROVE-WITH-NOTES | REJECT> — <notes>  (risky fixes only)

## Outcome
- Commit: <hash>
- Status: <committed | pending-approval | blocked>
- Date: <YYYY-MM-DD>

Siblings: `change.diff` (applied diff); `benchmark/` (micro-benchmark sources, if built).
```
